## 3. 第4章系统总体设计（定稿文本）

> 以下内容可直接作为论文正文替换/重写基础。

## 4.1 数据库设计

本系统采用 MySQL 8.0 作为关系型数据库管理系统，采用 Redis 作为缓存组件。数据库设计遵循“规范性、可扩展性、性能优先”原则，围绕用户、商品、拼团订单、自提点、评论等核心业务实体进行建模，支撑小程序端、团长工作台与后台管理端的统一数据协同。

## 4.1.1 数据库设计规范

1. 统一使用 `utf8mb4` 字符集与 `utf8mb4_unicode_ci` 排序规则，支持中文与特殊字符存储；  
2. 主键统一命名为 `id`，逻辑删除字段统一命名为 `deleted`（0未删除，1已删除）；  
3. 核心业务表统一包含 `create_time` 与 `update_time` 字段；  
4. 金额字段采用 `decimal(10,2)`，避免浮点精度问题；  
5. 高频查询字段建立索引（如 `user_id`、`product_id`、`pick_point_id`、`status`、`category_id`）；  
6. 对权限敏感业务采用“角色+数据归属”双重校验（如团长仅可操作绑定站点订单）。

## 4.1.2 核心数据表设计

### （1）用户表 `users`

用于存储小程序用户与后台角色信息，区分普通用户、团长、管理员/客服。

关键字段：
- `id bigint`：主键
- `openid varchar(100)`：微信唯一标识
- `nickname varchar(50)`：昵称
- `avatar varchar(255)`：头像
- `mobile varchar(20)`：手机号
- `is_leader tinyint(1)`：是否团长
- `admin_role varchar(20)`：后台角色（如 super_admin/leader/cs_agent）
- `create_time / update_time / deleted`

### （2）商品分类表 `product_categories`

用于维护商品分类字典并支持后台配置化管理。

关键字段：
- `id bigint`：主键
- `name varchar(50)`：分类名称（中文）
- `status tinyint(1)`：启用状态
- `sort int`：排序
- `create_time / update_time / deleted`

### （3）商品表 `products`

存储商品基础信息、价格与库存，关联分类。

关键字段：
- `id bigint`：主键
- `name varchar(100)`：商品名称
- `category_id bigint`：分类ID（关联 `product_categories.id`）
- `price decimal(10,2)`：单买价
- `group_price_2 decimal(10,2)`：2人团价
- `group_price_3 decimal(10,2)`：3人团价
- `stock int`：库存
- `images text`：商品图集（JSON）
- `status tinyint(1)`：上架状态
- `create_time / update_time / deleted`

### （4）自提点（站点）表 `pick_points`

存储提货站点信息并绑定团长，实现“团长仅管本点订单”。

关键字段：
- `id bigint`：主键
- `name varchar(100)`：站点名称
- `address varchar(255)`：站点地址
- `leader_name varchar(50)`：团长姓名（兼容字段）
- `leader_user_id bigint`：团长用户ID（强绑定）
- `phone varchar(20)`：联系电话
- `latitude/longitude double`：经纬度
- `create_time / update_time / deleted`

### （5）订单表 `orders`

订单核心业务表，记录下单、成团、核销、退款等状态流转。

关键字段：
- `id varchar(64)`：订单号
- `user_id bigint`：下单用户
- `product_id bigint`：商品ID
- `pick_point_id bigint`：自提点ID
- `group_buy_id varchar(64)`：拼团号
- `total_price decimal(10,2)`：实付金额
- `status tinyint(2)`：订单状态（1待成团，2已成团，3已取货，-1已取消/退款）
- `create_time / update_time / deleted`

### （6）商品评论表 `product_reviews`

用于商品真实评论，支撑详情页“买家评价”模块。

关键字段：
- `id bigint`：主键
- `product_id bigint`：商品ID
- `user_id bigint`：评论用户
- `content varchar(200)`：评论内容
- `rating tinyint`：评分（1-5）
- `create_time / update_time / deleted`

业务约束：仅存在该商品“已收货（status=3）”订单的用户可评论。

## 4.1.3 索引设计（建议写入论文）

- `orders(user_id, status, create_time)`：用户订单列表查询
- `orders(pick_point_id, status)`：团长工作台待核销查询
- `orders(group_buy_id, status)`：拼团进度查询
- `products(category_id, status)`：分类商品列表
- `product_reviews(product_id)`：详情页评论查询
- `pick_points(leader_user_id)`：团长站点权限判断

## 4.1.4 缓存设计（Redis）

- 缓存对象：热点商品列表、系统配置、部分统计数据；
- 更新策略：后台更新商品/配置后，主动清理对应缓存键；
- 目标：降低数据库读压力，提升小程序首屏与列表响应速度。

## 4.1.5 E-R 图设计说明（配图文字）

E-R 图应体现以下实体与关系：
- 用户（1）—（n）订单
- 商品分类（1）—（n）商品

- 自提点（1）—（n）订单

- 用户（1）—（n）自提点（通过 `leader_user_id` 体现团长绑定关系，业务上为团长对应站点）

## 6. 答辩高频追问与标准回答（可背）

### Q1：如何保证团长不能看其他站点订单？
A：后端不仅判断 `is_leader`，还校验 `pick_points.leader_user_id` 与当前登录团长 `user.id` 一致，不一致直接拒绝。

### Q2：评论是否可以刷？
A：当前版本限制“仅已收货订单可评”，即必须存在该用户该商品 `status=3` 的订单记录。

### Q3：为什么用 Redis？
A：用于热点读缓存，减少高频查询对 MySQL 的压力，提升首页和列表响应速度。

### Q4：为什么商品分类单独建表？
A：便于后台维护与扩展，避免硬编码分类文本导致的数据不一致与维护困难。
